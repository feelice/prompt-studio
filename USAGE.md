# 使用说明 / Usage Guide — Prompt Studio v0.2 beta

本文详细说明工具的使用方式，分为「全局使用」与「特定项目使用」两种场景，并针对程序开发、维护、测试给出应遵循的规范要求。

---

## 一、基本流程

1. 双击 `index.html`，浏览器本地打开（无需联网）。
2. 选「目标类型」：通用大模型 / 编程类 Agent / 自动化 Agent / 绘图 / 提示工程。
3. 选「场景模板」自动填充，或用「⤵ 导入并定制」粘贴自己的内容解析。
4. 编辑五个字段，替换其中 `{变量}`。
5. 选输出层级：简版 / 标准 / 详尽。
6. 复制 / 下载 `.md` / 复制含使用规范。

> 三档层级建议：**简版**用于熟悉场景的快速调用；**标准**用于日常；**详尽**用于复杂、高风险或新人协作任务（会自动加入「先复述理解、分步推理、完成自检」要求）。

## 二、全局使用 vs. 特定项目使用

同一个 prompt，按「作用范围」可分两层，理解两层能避免「规则到处写又互相打架」。

### 1. 全局（Global）

适用于你**所有对话 / 所有项目**都希望生效的稳定偏好（语气、语言、通用质量底线）。

| 工具 | 全局位置 | 说明 |
|---|---|---|
| 通用大模型(ChatGPT/Claude) | 自定义指令 / Project 级 system prompt | 角色、语气、默认语言与格式 |
| Cursor | Settings → Rules for AI | 跨所有项目的编码偏好 |
| Claude Code | `~/.claude/CLAUDE.md` | 个人级、跨仓库生效 |
| GitHub Copilot | 编辑器设置中的自定义指令 | 跨仓库通用指令 |

要点：只放**普适、稳定、与具体技术栈无关**的内容；不要放某项目的目录结构、框架版本；越短越好。

### 2. 特定项目（Project）

适用于**某个仓库 / 某个产品**特有的约定，优先级高于全局。

| 工具 | 项目位置 | 说明 |
|---|---|---|
| Cursor | `.cursor/rules/*.mdc`（新）/ `.cursorrules`（旧） | 随仓库提交，团队共享 |
| Claude Code | `./CLAUDE.md`、`./CLAUDE.local.md` | 项目级与本地覆盖 |
| 跨工具新标准 | `AGENTS.md` | 越来越多工具识别的通用 agent 说明 |
| GitHub Copilot | `.github/copilot-instructions.md` | 随仓库共享 |

要点：写清**技术栈与版本、目录约定、命名规范、运行/测试命令、不可触碰的部分**；用「编程类 Agent → Cursor / AGENTS 规则」模板生成骨架再填实情；随代码进版本库。

> **优先级**：项目规则 > 全局规则。冲突以项目为准；全局只写「最大公约数」，项目写「差异化细节」。

## 三、程序开发 / 维护 / 测试 应遵循的规范

建议把以下要点写进「约束条件」字段或项目规则文件。

### 1. 开发（Development）
- **改前先读**：修改前先阅读相关代码与现有约定，禁止凭空假设。
- **不臆造**：不虚构 API / 函数签名 / 配置项；不确定就说明并停下询问。
- **小步改动**：一次只解决一个问题，最小化 diff，便于审查与回滚。
- **复用优先**：优先复用已有工具函数/组件。
- **依赖管控**：不擅自引入新依赖；确需引入先说明理由并征得同意。
- **风格一致**：遵循格式化、命名、Lint 与类型检查，提交前必须通过。
- **先方案后执行**：较大任务先给方案与影响范围，确认后再动手。
- **安全底线**：不硬编码密钥；注意输入校验、注入与越权；敏感数据不外泄。

### 2. 维护（Maintenance）
- **先定位根因**：修 Bug 先复现并找根因，避免「头痛医头」。
- **防回归**：不破坏既有行为；公共接口保持向后兼容或给迁移说明。
- **可追溯**：说明「改了什么、为什么、有何副作用」；遵循 **Conventional Commits**（`type(scope): subject`）。
- **重构守则**：保持外部行为不变，小步提交，每步可验证，补足测试。
- **文档同步**：行为/接口变化时同步更新 README / 注释 / 变更日志。

### 3. 测试（Testing）
- **覆盖三类路径**：正常、边界、异常/错误。
- **结构清晰**：AAA（Arrange–Act–Assert）；用例独立、命名见名知意。
- **确定性**：结果稳定可复现，不依赖时间/网络/随机（必要时 mock）。
- **修 Bug 先加测试**：先写复现失败用例再修复，确保不回归。
- **合并门槛**：测试、Lint、类型检查、构建全过（CI 绿）才合并。
- **有效覆盖**：追求关键逻辑被覆盖，而非堆覆盖率数字。

## 四、把规范落到本工具

1. 选「编程类 Agent」→「Cursor / AGENTS 规则」，或在「提示工程」里选「系统提示词构建器」。
2. 在「约束与验收」字段粘贴上面对应的开发/维护/测试条目。
3. 选「详尽」层级生成 → 下载 `.md`。
4. 全局规范放全局位置；项目规范保存为 `.cursor/rules/`、`CLAUDE.md` 或 `AGENTS.md`，随仓库提交。

这样 AI 在每次开发、维护、测试时都会自动遵循同一套规范。
